# Avortyx Backend API Contract

> **Purpose.** This document describes every HTTP and WebSocket request the
> Avortyx frontend currently sends. It is generated by reading the source of
> truth (`lib/api/services/*.ts`, `lib/api/http.ts`, `lib/api/socket.ts`) and
> is intended for the backend team as the canonical reference when wiring up
> endpoints.
>
> **All endpoint paths in this document are the literal strings the frontend
> issues.** Trailing slashes, query-string keys, and body field names are
> documented exactly as the wire sees them after the case-conversion middleware
> runs (see §3.1). If something in this document does not match the backend's
> actual behavior, the gap is a contract mismatch that needs to be reconciled.

## 1. Environment & base URLs

The frontend reads two environment variables at build time (Next.js inlines
`NEXT_PUBLIC_*` literals into the client bundle):

| Env var | Purpose | Example |
| --- | --- | --- |
| `NEXT_PUBLIC_API_BASE_URL` | Origin of the REST API. **No trailing slash.** | `https://api.avortyx.io` |
| `NEXT_PUBLIC_WS_BASE_URL` | Origin of the WebSocket channel. **No trailing slash.** | `wss://api.avortyx.io` |

A literal trailing `/` is stripped at startup. Every request path in this
document is appended directly to the base URL.

## 2. Authentication

### 2.1 Token model

The frontend uses a two-token JWT scheme:

- **Access token** — short-lived, sent on every authenticated request as
  `Authorization: Bearer <access>`.
- **Refresh token** — long-lived, used only against `POST /api/accounts/refresh`
  to mint a new access token.

Both are persisted to `localStorage` under the key `avortyx.tokens` (shape
`{ "access": "...", "refresh": "..." }`).

### 2.2 Refresh-on-401 retry

When any non-anonymous request returns **HTTP 401**, the HTTP client:

1. Calls `POST /api/accounts/refresh` exactly once (concurrent 401s coalesce
   into a single refresh).
2. If refresh succeeds (response `{ "access": "..." }`, status 2xx), the
   original request is retried once with the new access token.
3. If refresh fails (non-2xx) or returns no `access`, local tokens are cleared
   and the original 401 propagates to the caller.

The refresh request itself is never retried; the refresh endpoint must NOT
require an `Authorization` header.

### 2.3 Anonymous endpoints

These endpoints are issued without an `Authorization` header even if a token
is present in localStorage:

- `POST /api/accounts/login`
- `POST /api/accounts/register`
- `POST /api/accounts/refresh`
- `POST /api/accounts/password-reset/request`
- `POST /api/accounts/password-reset/confirm`
- `POST /api/accounts/access-requests/`
- `POST /api/accounts/set-password/`
- `POST /api/support/chat`
- `POST /api/support/chat/{session_id}`
- `GET  /api/support/chat/{session_id}`

All other endpoints assume a bearer token; if absent or invalid, the backend
should respond 401 and the client will attempt a refresh as described above.

## 3. HTTP conventions

### 3.1 Case conversion

The HTTP client performs **bidirectional case conversion** at the wire boundary:

- **Outgoing**: every key in request bodies (JSON) and query strings is
  converted from `camelCase` to `snake_case` before being sent. Example: the
  frontend writes `{ phoneNumber: "+15551234567" }`, the wire carries
  `{ "phone_number": "+15551234567" }`.
- **Incoming**: every key in JSON response bodies is converted from
  `snake_case` to `camelCase` before being handed to the frontend. Example:
  the wire returns `{ "created_at": "2026-06-23T..." }`, the frontend reads
  `createdAt`.

**Implication for the backend:** all field and query-parameter names in this
document are written in `snake_case` because that is what the wire carries.
The "camelCase" names that appear in the frontend code (e.g. `campaignId`)
arrive as `campaign_id` on the wire and vice versa.

The conversion skips `Date`, `File`, `Blob`, `FormData`, `ArrayBuffer`, and
`ArrayBufferView` values. `FormData` bodies are sent verbatim with no JSON
serialization and no `Content-Type` header (the browser sets the multipart
boundary).

### 3.2 Trailing slashes — inconsistent across resources

The frontend does **not** normalize trailing slashes. Each request hits the
exact path documented per endpoint. The current state of the codebase is
inconsistent across resources — the backend must accept the literal path the
frontend sends:

| Resource | List path | Detail path |
| --- | --- | --- |
| `/api/campaigns` | `/api/campaigns/` (trailing) | `/api/campaigns/{id}` (no trailing) |
| `/api/numbers` | `/api/numbers/` (trailing) | `/api/numbers/{id}` (no trailing) |
| `/api/dni/pools` | `/api/dni/pools` (no trailing) | `/api/dni/pools/{id}` (no trailing) |
| `/api/destinations` | `/api/destinations/` (trailing) | `/api/destinations/{id}/` (trailing) |
| `/api/buyers` | `/api/buyers/` (trailing) | `/api/buyers/{id}` (no trailing) |
| `/api/publishers` | `/api/publishers/` (trailing) | `/api/publishers/{id}` (no trailing) |
| `/api/spam/blacklist` | `/api/spam/blacklist` (no trailing) | `/api/spam/blacklist/{id}` (no trailing) |
| `/api/spam/whitelist` | `/api/spam/whitelist` (no trailing) | `/api/spam/whitelist/{id}` (no trailing) |
| `/api/spam/shields` | `/api/spam/shields/` (trailing) | `/api/spam/shields/{id}/` (trailing) |
| `/api/webhooks` | `/api/webhooks/` (trailing) | `/api/webhooks/{id}` (no trailing) |
| `/api/webhooks/pixels` | `/api/webhooks/pixels/` (trailing) | `/api/webhooks/pixels/{id}` (no trailing on GET, trailing on PATCH/DELETE) |
| `/api/analytics/reports` | `/api/analytics/reports/` (trailing) | `/api/analytics/reports/{id}/` (trailing) |
| `/api/notifications/rules` | `/api/notifications/rules` (no trailing) | `/api/notifications/rules/{id}` (no trailing) |
| `/api/accounts/api-keys` | `/api/accounts/api-keys/` (trailing) | `/api/accounts/api-keys/{id}` (no trailing) |
| `/api/accounts/access-requests` | `/api/accounts/access-requests/` (trailing) | `/api/accounts/access-requests/{id}/{action}/` (trailing) |
| `/api/ai/recommendations` | `/api/ai/recommendations/` (trailing) | — |
| `/api/ai/autopilot/config` | `/api/ai/autopilot/config/` (trailing) | — |
| `/api/ai/anomalies` | `/api/ai/anomalies/` (trailing) | — |
| `/api/white-label` | `/api/white-label/` (trailing) | — |
| `/api/referrals` | `/api/referrals/` (trailing on root) | mixed (see §7.25) |

If the backend's URL router uses `APPEND_SLASH = True` (Django default), all
non-trailing variants will be 301-redirected to the trailing form which the
frontend follows transparently. If `APPEND_SLASH = False`, the literal path
must be served. The exact literal paths used by the frontend are documented
per endpoint in §7.

### 3.3 Idempotent POST de-duplication

Identical authenticated POST requests fired concurrently (same path, same
body) are coalesced into a single network call. The fingerprint key is
`<path>::<JSON.stringify(body)>`. Anonymous POSTs and `FormData` POSTs are
exempt. This is a client-side guard against double-clicks; the backend does
not need to support `Idempotency-Key` headers.

### 3.4 Request options

The frontend supports cancellation via `AbortSignal`. Long-running responses
should honor connection close cleanly.

### 3.5 Response decoding

- **HTTP 204 No Content** → returns `null` to callers without attempting to
  parse a body.
- **`Content-Type: application/json`** → parsed and case-converted.
- **Non-JSON Content-Type** → returned as raw text (used for CSV export).
- **Empty body** → returns `null`.

## 4. Error envelope

Any non-2xx response is normalized into an `ApiError` with `status`,
`message`, optional `code`, and optional `fieldErrors[]`. The client accepts
three shapes for the error body:

### 4.1 Django Ninja validation (preferred)

```json
{
  "detail": [
    { "loc": ["body", "email"], "msg": "Enter a valid email address.", "type": "value_error.email" },
    { "loc": ["body", "password"], "msg": "Must be at least 8 characters." }
  ]
}
```

The frontend strips the leading `body` / `query` / `path` segment from each
`loc` and builds a `fieldErrors[]` array of `{ field, message }`. The
user-facing top-line becomes `"<field_names>: <first message>"`.

### 4.2 Single-message detail

```json
{ "detail": "Workspace name already in use." }
```

### 4.3 Custom `{ message, code }`

```json
{ "message": "Insufficient balance.", "code": "billing.insufficient_funds" }
```

### 4.4 Status-code fallbacks

If the body is empty, HTML (Django debug page), or a Python traceback, the
frontend substitutes a per-status user-facing string:

| Status | User message |
| --- | --- |
| 400 | Request couldn't be processed. Please double-check the values and try again. |
| 401 | Your session has expired. Please sign in again. |
| 403 | You don't have permission to do that. |
| 404 | We couldn't find what you were looking for. |
| 409 | This action conflicts with the current state. Please refresh and try again. |
| 422 | Some fields are missing or invalid. Please review and try again. |
| 429 | Too many requests — please wait a moment and try again. |
| 5xx | Something went wrong on our end. Please try again in a moment. |

Backends should prefer §4.1 or §4.2 when they want a specific message
displayed to the user.

## 5. Pagination

### 5.1 Canonical envelope

Every list endpoint (with the exception of `/api/analytics/calls`,
documented separately in §7.17) should respond with:

```json
{
  "items":      [/* ... */],
  "total":      123,
  "page":       1,
  "page_size":  50
}
```

After case conversion the frontend reads `{ items, total, page, pageSize }`.

### 5.2 Query parameters

The frontend sends:

| Wire key | Type | Meaning |
| --- | --- | --- |
| `page` | int (1-based) | Page number, defaults to 1 if omitted. |
| `page_size` | int | Items per page. Defaults vary by surface (25, 50, 100). |

### 5.3 Tolerant envelope parser

The spam blacklist/whitelist/reports/shields, marketplace, queue, IVR flows,
workspace members, API keys, referred clients, and publisher payouts
endpoints use a **tolerant envelope parser** that accepts:

- `{ items, total, page, page_size }` (canonical)
- `{ results, count, next, previous }` (Django REST Framework default)
- `{ data: [...] }`
- a bare `[...]` array

This tolerance is a temporary safety net; new endpoints should ship the
canonical envelope in §5.1.

### 5.4 The `/api/analytics/calls` exception

The Call Log endpoint uses `offset` / `limit` query params (not
`page` / `page_size`) and responds with:

```json
{
  "total":  1234,
  "offset": 0,
  "limit":  50,
  "items":  [/* ... */]
}
```

The frontend translates page/pageSize into offset/limit at the service
boundary. Documented in detail in §7.17.

## 6. WebSocket channel

### 6.1 Endpoint

```
{WS_BASE_URL}/ws/live-calls/?token=<access_token>
```

- **Single channel** for both call lifecycle events and RTB auction/bid
  events.
- **JWT auth** via the `token` query parameter (URL-encoded). Sub-protocol
  headers are NOT used.
- The connection is dropped if the token is missing, expired, or invalid;
  the client reconnects with exponential backoff once a new token is
  available (1 s → 30 s cap).
- The client pauses reconnect attempts while `document.visibilityState` is
  `"hidden"` and resumes on visibility change.

### 6.2 Message envelope

Each frame the server sends must be a JSON object of the form:

```json
{ "type": "call.connected", "data": { /* event payload */ } }
```

The `type` field is a dotted string — case conversion does **not** alter
string values, only object keys. `data` is an arbitrary object whose keys
will be snake→camel converted on receipt.

### 6.3 Canonical event types

```
call.created
call.ringing
call.connected
call.ended
call.routed
call.failed
call.queued
call.spam_blocked
rtb.auction_created
rtb.bid_placed
rtb.auction_settled
```

The frontend dispatches handlers by `type`. Events with unknown types are
silently dropped. Add a new type only after coordinating with the frontend
team to add it to the `CALL_EVENT_TYPES` constant.

### 6.4 `call.*` event data shape

`data` should match the `CallRecord` wire shape documented in §7.16 / §7.17
(snake_case on the wire). The frontend reuses the same mapper for REST
responses and socket payloads.

### 7.1 Accounts & auth

Base path: `/api/accounts`

#### `POST /api/accounts/login` *(anonymous)*

Request body:
```json
{ "email": "string", "password": "string" }
```

Response body:
```json
{
  "access":          "string",
  "refresh":         "string",
  "user_id":         "string (uuid)",
  "email":           "string",
  "role":            "string",
  "organization_id": "string (uuid)"
}
```

The frontend immediately persists `access` + `refresh` to localStorage and
then calls `GET /me` to hydrate the full profile.

#### `POST /api/accounts/register` *(anonymous)*

Request body:
```json
{
  "email":             "string",
  "password":          "string",
  "first_name":        "string",
  "last_name":         "string",
  "organization_name": "string"
}
```

Response: any 2xx accepted; body is ignored. The frontend immediately calls
`/login` with the same credentials to obtain tokens.

If the user supplied a phone during signup, the frontend follows up with
`PATCH /api/accounts/me { "phone_number": "..." }`. Failures here are
swallowed (phone is decorative).

#### `POST /api/accounts/refresh` *(anonymous)*

Request body:
```json
{ "refresh": "string" }
```

Response body:
```json
{ "access": "string" }
```

A response without an `access` string of length > 0, or a non-2xx status,
causes the frontend to clear tokens and force re-login.

#### `POST /api/accounts/logout`

No request body. Any 2xx accepted; local tokens are dropped regardless of the
server's response.

#### `GET /api/accounts/me`

No query, no body.

Response body:
```json
{
  "id":                "string (uuid)",
  "email":             "string",
  "first_name":        "string",
  "last_name":         "string",
  "role":              "string",
  "phone_number":      "string?",
  "mfa_enabled":       "boolean?",
  "is_email_verified": "boolean?",
  "is_superuser":      "boolean?",
  "organization_id":   "string (uuid)?",
  "avatar_url":        "string?",
  "organization_name": "string?"
}
```

#### `PATCH /api/accounts/me`

Request body — any subset of:
```json
{
  "first_name":   "string?",
  "last_name":    "string?",
  "phone_number": "string?",
  "avatar_url":   "string?"
}
```

Response body: same shape as `GET /api/accounts/me`.

#### `POST /api/accounts/me/avatar`

`multipart/form-data` with a single field:

| Field | Type | Notes |
| --- | --- | --- |
| `file` | binary | The image to upload. |

Response body: same shape as `GET /api/accounts/me`.

#### `POST /api/accounts/change-password`

Request body:
```json
{ "current_password": "string", "new_password": "string" }
```

#### `POST /api/accounts/password-reset/request` *(anonymous)*

Request body:
```json
{ "email": "string" }
```

#### `POST /api/accounts/password-reset/confirm` *(anonymous)*

Request body:
```json
{ "token": "string", "new_password": "string" }
```

### 7.2 Access requests (invite-only signup)

Base path: `/api/accounts/access-requests`

The flow: prospect submits a request → admin approves → backend creates the
user without a password and emails a `setup_link` containing a one-time
token → user submits a new password.

#### `POST /api/accounts/access-requests/` *(anonymous, rate-limited)*

Request body:
```json
{
  "name":     "string",
  "company":  "string",
  "email":    "string",
  "phone":    "string",
  "use_case": "string"
}
```

Response body — an `AccessRequest` (see below).

#### `GET /api/accounts/access-requests/`

Admin only. Optional query: `?status=pending|approved|rejected`.

Response — tolerant envelope (§5.3) of `AccessRequest` objects:

```json
{
  "id":                "string (uuid)",
  "name":              "string",
  "company":           "string",
  "email":             "string",
  "phone":             "string",
  "use_case":          "string",
  "status":            "pending|approved|rejected",
  "created_at":        "iso-8601",
  "reviewed_at":       "iso-8601?",
  "reviewed_by":       "string?",
  "rejection_reason":  "string?"
}
```

#### `POST /api/accounts/access-requests/{id}/approve/`

Request body:
```json
{ "role": "string" }
```

`role` defaults to `"buyer"` on the frontend. Backend must create the user
and send the setup email synchronously before responding. Response: an
`AccessRequest` with `status: "approved"`.

#### `POST /api/accounts/access-requests/{id}/reject/`

Request body (reason optional):
```json
{ "reason": "string?" }
```

Response: an `AccessRequest` with `status: "rejected"`.

#### `POST /api/accounts/set-password/` *(anonymous)*

Request body:
```json
{ "token": "string", "password": "string" }
```

Response — same shape as `POST /api/accounts/login` so the frontend can sign
the user in immediately.

### 7.4 API keys

Base path: `/api/accounts/api-keys`

Keys are scoped to the caller's organization via the bearer token; no org id
is sent on the wire. The plaintext secret is returned **once**, on create,
and never afterwards.

#### `GET /api/accounts/api-keys/`

Response — bare array OR `{ items: [...] }` of:
```json
{
  "id":              "string (uuid)",
  "name":            "string",
  "prefix":          "string?",          // any of these three holds the
  "key_prefix":      "string?",          // masked display value
  "masked_key":      "string?",
  "scopes":          ["read"|"write"|"admin", ...]?,
  "permissions":     [...]?,             // alias for scopes
  "created_at":      "iso-8601?",
  "last_used_at":    "iso-8601?",
  "created_by_name": "string?",
  "created_by":      "string?"
}
```

#### `POST /api/accounts/api-keys/`

Request body:
```json
{ "name": "string", "scopes": ["read"|"write"|"admin", ...] }
```

Response — the key record above **plus** one of these fields holding the
one-time plaintext secret (frontend reads whichever is present):

```
key | token | secret | full_key | plaintext | api_key
```

#### `DELETE /api/accounts/api-keys/{id}`

Returns 204 on success.

**Rotate.** There is no backend rotate endpoint; the frontend composes
rotation as `create` → `delete` on the old id.

### 7.7 Campaigns

Base path: `/api/campaigns`

The frontend uses long human-readable labels for vertical and a single
schedule block; both are translated at the wire boundary.

#### Enum maps

**Payout model** (frontend → wire):
| FE | Wire |
| --- | --- |
| `per-call` | `per_call` |
| `per-qualified` | `per_qualified` |
| `per-minute` | `per_minute` |

**Vertical** (frontend → wire):
| FE label | Wire enum |
| --- | --- |
| Health Insurance | `health` |
| Auto Insurance | `insurance` |
| Automotive | `auto` |
| Home Services | `home` |
| Finance | `finance` |
| Legal | `legal` |
| Insurance | `insurance` |
| Other | `other` |

**Weekday convention.** The frontend uses 0=Sunday..6=Saturday;
**the backend uses 0=Monday..6=Sunday.** Conversion happens at the wire
boundary.

**Time format.** The backend's `POST` schedule validator only accepts
`"HH:MM"` (NOT `"HH:MM:SS"`). Sending seconds returns the validator error
"Time must be in HH:MM format". Hours are clamped to 0–23. The end-of-day
sentinel `endHour: 24` is sent as `close_time: "23:00"`.

```json
  "vertical":       "string?",
  "payout_model":   "string?"
}
```

```json
    "max_calls_monthly":  "int?",
    "max_calls_global":   "int?",
    "max_concurrency":    "int?"
  },
  "schedules": [
    {
      "id":           "string (uuid)?",
      "day_of_week":  "int (0=Mon..6=Sun)",
      "open_time":    "HH:MM[:SS]",
      "close_time":   "HH:MM[:SS]",
```

#### `POST /api/numbers/{id}/assign`

Request body:

### 7.20 Shields (VoIP + TCPA)

```json

  "shield_type":      "voip|tcpa",
  "campaign_ids":     ["string (uuid)", ...]?,
  "blocked_carriers": ["string", ...]?,
  "is_active":        "boolean?"
}
```

#### `GET /api/spam/shields/{id}/`

Single shield.

#### `POST /api/spam/shields/`

Request body:
```json
{
  "name":             "string",
  "shield_type":      "voip|tcpa",
  "campaign_ids":     ["string (uuid)", ...],   // defaults to []
  "blocked_carriers": ["string", ...],          // defaults to []
  "is_active":        "boolean"                 // defaults to true
}
```

#### `PATCH /api/spam/shields/{id}/`

Request body — any subset of:
```json
{
  "name":             "string?",
  "campaign_ids":     ["string (uuid)", ...]?,
  "is_active":        "boolean?",
  "blocked_carriers": ["string", ...]?
}
```

**OPEN CONTRACT ISSUE.** The backend dev documented PATCH allowlist as
`name`, `campaign_ids`, `is_active`. The frontend also sends
`blocked_carriers` because that field is the sole transport for:

- the VoIP Shield's blocked-carrier list (rendered as a list of strings)
- the TCPA Shield's per-provider config (frontend packs `type`, `active`,
  `config` into a single JSON-tagged string with the prefix
  `__tcpa_meta__|` and stores it as a single-element array)

If the backend silently drops `blocked_carriers` from updates, both flows
appear to save but immediately revert. **Please add `blocked_carriers` to
the PATCH-allowed field list.** See §8.

## 8. Known open contract issues

These are documented because the frontend has either worked around them or
surfaces them as visible UI bugs. Resolving them is a backend ask.

### 8.1 `PATCH /api/spam/shields/{id}/` silently drops `blocked_carriers`

**Symptom.** VoIP Shield "Block Carrier" updates and TCPA Shield provider
config (login/password/timeout/active) appear to save successfully — the
response is 200 OK — but the field is unchanged in the response body. The
frontend's response-reconcile pattern then reverts the UI to the
backend-persisted value.

**Affected flows.**
- VoIP Shield → blocked-carriers list
- TCPA Shield → provider config (packed into `blocked_carriers` as a single
  JSON-tagged string with prefix `__tcpa_meta__|`)
- TCPA Shield → active toggle on the list page (packed into the same blob)

**Ask.** Add `blocked_carriers` (string array) to the PATCH-allowed field
list on the shield serializer alongside `name`, `campaign_ids`, `is_active`.

### 8.2 `PATCH /api/numbers/{id}` silently drops new fields

**Symptom.** Editing a tracking number's Name (`label`) or Allocated
(`allocated_capacity`) returns 200 OK but the response carries the old
values. UI reverts immediately via the response-reconcile pattern.

**Confirmed accepted fields.** `campaign_id`, `status`.

**Ask.** Add to PATCH-allowed fields:
- `label` (string)
- `allocated_capacity` (int)
- `cap_enabled` (boolean)
- `daily_cap` (int)

### 8.3 `DELETE /api/campaigns/{id}` silently no-ops

**Symptom.** DELETE returns 2xx but the campaign row remains in the
database. The frontend's optimistic removal therefore reverts on the next
list refresh.

**Ask.** Investigate why DELETE doesn't actually delete the row. The
canonical URL is **without** a trailing slash (the trailing-slash variant
404s).

### 8.4 Trailing-slash inconsistency

The frontend hits both `/api/...` and `/api/.../` variants depending on the
resource (see §3.2 for the full per-resource table). Decision: either
configure the backend with `APPEND_SLASH = True` so both variants resolve,
or pin every resource to the same convention and update the frontend
accordingly. The current behavior must be preserved until a coordinated
sweep happens.

### 8.5 Field-name immutability needs clear errors

Where a field is intentionally read-only (`phone_number` on spam
blacklist/whitelist updates; `organization_name` / `organization_id` on
buyers/publishers create+update), the backend should respond with a clear
`{ detail: [{loc: [...], msg: "..."}] }` validation error if the field is
sent. Silently accepting and ignoring those fields hides bugs.
